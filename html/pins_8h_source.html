8 /*
9 The tables declared here identify compatible pins and their corresponding
10 SERCOMs and DMA-related registers and such. SAMD21/51 can actually handle
11 SPI DMA on many more pins than are indicated here, but the code design
12 INTENTIONALLY limits it to specific pins -- one "curated" pin per SERCOM.
13 Not for technical reasons, but as a matter of documentation and clarity.
14 Although each SERCOM could work with a choice of MOSI pins, it becomes
15 extremely verbose to try to explain "if you choose pin X for output, then
16 you can't use pins Y or Z" (and then repeating this for every SERCOM,
17 times every supported board). It's INFINITELY SIMPLER to explain and use
18 if one "good pin" has been preselected per SERCOM for each board. The user
19 then has a list of valid pins for any given board, they can use any one
20 (or more, if multiple Adafruit_NeoPixel_ZeroDMA instances) in whatever
21 combination, no complicated if/else/else/else explanations needed.
22 I tried to pick pins that are nicely spaced around the board and don't
23 knock out other vital peripherals. SERCOM pin selection is NOT a fun
24 process, believe me, it's SO MUCH EASIER this way. Most programs will be
25 using only one output anyway, maybe a couple (if you need lots, consider
26 the NeoPXL8 library instead).
27 */

326 // NOTE TO FUTURE SELF: avoiding pins used by essential SERCOM peripherals
327 // (I2C, Serial) was a design decision to make documentation easier (not
328 // having to explain "if you have DMA NeoPixels on this pin, you can't use
329 // I2C devices") (exception being the SPI MOSI pin, because the older
330 // library handled that on Metro boards and it's not as widely used as I2C).
331 // HOWEVER, on many boards where there's only an "external" I2C bus (no
332 // onboard sensors or such sharing the bus), it mmmmight be sensible to
333 // allow DMA NeoPixels on either the SDA or SCL pins, since the NeoPixels
334 // at that point physically block I2C (ditto for the Serial1 TX/RX pins) --
335 // it's sort of implied that the peripheral can't be used at the same time,
336 // but as implemented right now, it additionally *enforces* not using DMA
337 // NeoPixels on those pins (at all, not just when not-using-peripheral).
338 // Maybe that's too strict and not necessary. Or maybe the selection of
339 // pins here, as-is, adequately covers most situations. Just saying there
340 // might be a possibility of having to revisit these tables to add 1-2 more
341 // pin options that overlap I2C or Serial1 on boards where those are
342 // physically exposed and not shared with onboard peripherals. It's no fun,
343 // requires using the pinfinder.py script (in extras directory) and looking
344 // for the right missing items to add on a per-board basis.
345 // (There's a couple of exceptions in the lists, if a board just has no
346 // other SPI-MOSI-DMA-capable pins, one is exposed and will require an
347 // asterisk in the docs, that it takes out some other peripheral.)
348

352 // 1) not an existing SERCOM that's used for a vital peripheral (I2C,
353 // Serial1 or SPI, or in the case of certain boards with onboard sensors
354 // and a second internal peripheral bus, avoid that bus), and...
355 // 2) on PAD[0] or PAD[3], or if it's a SAMD21 part, then PAD[2] also.
